# LangFlow: QA Knowledge Base Chatbot (Simple RAG)

## Problem Statement

Build a `LangFlow` workflow that ingests a QA-related document and answers user questions using retrieval-augmented generation (`RAG`).

The chatbot should answer only from the uploaded document instead of relying on general model knowledge.

## Context

QA teams often store important information inside:

- test plans
- PRDs
- release notes
- API documentation
- validation checklists

Searching those files manually is slow. This flow turns the uploaded document into a searchable knowledge base so a tester can ask questions like:

- `What are the entry criteria for regression?`
- `Which endpoints require authentication testing?`
- `What is the expected login validation behavior?`

## Workflow Requirements

1. Use `Chat Input` for the user question.
2. Use a `File` input to ingest the QA document text.
3. Use `Recursive Character Text Splitter` with:
   - `chunk_size = 500`
   - `chunk_overlap = 50`
4. Use an `Embeddings` component.
5. Use a vector store such as `ChromaDB` or `FAISS`.
6. Use this exact prompt template:

```text
Answer the question based ONLY on the context below.
Context: {context}
Question: {question}
```

7. Send the answer to `Chat Output`.

## Practical Build Choice

The most reliable local build for this repo uses:

```text
Chat Input
Read File
Type Convert
Recursive Character Text Splitter
OpenAI Embeddings
Chroma DB
Parser
Prompt Template
Language Model
Chat Output
```

Why this version:

- it matches the actual challenge requirement more closely than the default starter
- it uses `Chroma DB`, which works well for a simple local RAG demo
- it uses `Type Convert` so the uploaded file output can be passed safely into the recursive splitter
- it stays importable without requiring external vector DB services

## Node-by-Node Design

### 1. Read File

Use `Read File` to upload a QA document such as:

- `test-plan.md`
- `prd.txt`
- `api-spec.md`

This is the knowledge source for the chatbot.

### 2. Type Convert

Use `Type Convert` to convert the file output from `Message` to `Data`.

Set:

- `Output Type = Data`
- `Auto Parse = false`

This makes the file content compatible with the recursive splitter.

### 3. Recursive Character Text Splitter

Use `Recursive Character Text Splitter`.

Set:

- `Chunk Size = 500`
- `Chunk Overlap = 50`

This breaks the document into smaller overlapping chunks so retrieval is more precise.

### 4. OpenAI Embeddings

Use `OpenAI Embeddings`.

Suggested model:

- `text-embedding-3-small`

This converts each chunk into vectors that the vector store can search.

### 5. Chroma DB

Use `Chroma DB` as the vector store.

Suggested values:

- `Collection Name = qa_kb_docs`
- `Persist Directory = ./langflow_chroma/qa_kb_docs`
- `Search Type = Similarity`
- `Number of Results = 4`
- `Allow Duplicates = false`

This stores the embedded chunks and retrieves the most relevant ones for each question.

### 6. Parser

Use `Parser` on the retrieved `DataFrame`.

Suggested template:

```text
{text}
```

This turns the retrieved rows into a clean context block for the prompt.

### 7. Prompt Template

Use the following exact prompt:

```text
Answer the question based ONLY on the context below.
Context: {context}
Question: {question}
```

This reduces hallucination and keeps the response grounded in the retrieved chunks.

### 8. Language Model

Use `Language Model`.

Recommended local-friendly provider choice in this export:

- `OpenAI`

Recommended model:

- `gpt-4o-mini`

You can swap this later to `Ollama` if you want a local model instead.

### 9. Chat Output

Send the final response to `Chat Output`.

## Example Input

Question:

```text
What are the login page test scenarios?
```

Uploaded document contains requirements such as:

- valid login
- invalid credentials
- remember me
- input validation

## Example Output

```text
Based on the uploaded QA document, the login page test scenarios include valid login with correct credentials, invalid login with incorrect credentials, validation for empty or malformed input fields, and verification of the remember me option if enabled.
```

## Why This Output Works

- it is grounded in retrieved chunks from the uploaded document
- it does not invent functionality outside the provided context
- it is short, usable, and appropriate for QA analysis

## Evaluation Criteria Mapping

### 1. Approach

The flow uses the correct RAG pipeline:

- ingest document
- split into chunks
- embed chunks
- store vectors
- retrieve relevant context
- answer with prompt-grounded generation

### 2. Working Flow

The exported flow is structured so the uploaded file is converted, chunked, embedded, indexed in `Chroma`, retrieved by question, and passed through a prompt to the model.

### 3. Output Quality

The prompt explicitly says:

- answer only from context
- use retrieved text
- avoid unsupported claims
