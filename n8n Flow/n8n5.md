# LangFlow: API Test Case Generator with Prompt Chaining

## Problem Statement

Build a `LangFlow` workflow that takes API endpoint documentation as input, first extracts testable requirements, and then generates structured API test cases from those requirements.

This is a two-step prompt chaining workflow rather than a single large prompt.

## Context

QA engineers often receive endpoint documentation in plain text and need to convert it into useful test cases quickly.

Examples of source material include:

- API docs
- endpoint descriptions
- request and response rules
- validation constraints
- status code expectations

Instead of asking one model call to do everything at once, this workflow separates the reasoning into:

1. requirement extraction
2. test case generation

That makes the output easier to control and reduces invented behavior.

## Workflow Requirements

1. Use `Chat Input` where the user pastes API endpoint documentation.
2. Use a first LLM step with this instruction:

```text
You are an API analyst. Extract all testable requirements from this API documentation. List each requirement as a numbered item.
```

3. Use a second prompt template with this instruction:

```text
You are a Senior API Tester. Generate test cases for each requirement. Format: Test ID | Requirement | Method | Input | Expected Status | Expected Response.
```

4. Add these constraints:

```text
Cover happy path, negative scenarios, and boundary conditions. Do NOT invent undocumented behavior.
```

5. Connect the final result to `Chat Output`.

## Practical Build Choice

The cleanest local build for this repo uses:

```text
Chat Input
Prompt
Language Model
Prompt
Language Model
Chat Output
```

Why this version:

- it directly matches the two-step chaining requirement
- it is easy to test in the LangFlow playground
- it keeps the first step focused on extraction and the second step focused on structured generation
- it avoids unnecessary retrieval or tool complexity

## Recommended LangFlow Graph

```text
Chat Input
  -> Extract Requirements Prompt
  -> Requirements LLM
  -> Generate Test Cases Prompt
  -> Test Cases LLM
  -> Chat Output
```

Expanded chain:

```text
Chat Input
  -> Requirements LLM

Extract Requirements Prompt
  -> Requirements LLM system message

Requirements LLM output
  -> Generate Test Cases Prompt requirements input

Chat Input
  -> Generate Test Cases Prompt api documentation input

Generate Test Cases Prompt
  -> Test Cases LLM input

Test Cases LLM
  -> Chat Output
```

## Node-by-Node Design

### 1. Chat Input

Use `Chat Input` for the API endpoint documentation.

Suggested sample input:

```text
POST /api/users - Creates a new user. Required fields: name (string, 2-50 chars), email (valid email format), password (min 8 chars, must include uppercase and number). Returns 201 on success, 400 on validation error, 409 if email exists.
```

### 2. Extract Requirements Prompt

Use a `Prompt` node for the first step.

Recommended prompt:

```text
You are an API analyst.

Extract all testable requirements from this API documentation.

Rules:
- List each requirement as a numbered item.
- Use only the provided API documentation.
- Do not invent undocumented rules.
- Include request validation rules, required fields, boundary limits, method behavior, and status code expectations when present.
```

This prompt is connected to the first `Language Model` as the system instruction.

### 3. Requirements LLM

Use the first `Language Model` node to extract testable requirements.

Recommended settings:

- `Provider = OpenAI`
- `Model = gpt-4o-mini`
- `Temperature = 0.1`

Input:

- API documentation from `Chat Input`

Output:

- a numbered requirements list

### 4. Generate Test Cases Prompt

Use a second `Prompt` node for the test case generation step.

This node should receive:

- extracted requirements from step 1
- original API documentation from `Chat Input`

Recommended prompt:

```text
You are a Senior API Tester.

Generate test cases for each requirement.

Constraints:
- Cover happy path, negative scenarios, and boundary conditions.
- Do NOT invent undocumented behavior.
- If information is missing, write Needs clarification instead of guessing.
- Use the original API documentation only as supporting context.

Original API Documentation:
{question}

Extracted Requirements:
{context}

Format every row exactly as:
Test ID | Requirement | Method | Input | Expected Status | Expected Response
```

In this export:

- `{context}` is populated from the first LLM output
- `{question}` is populated from the original `Chat Input`

### 5. Test Cases LLM

Use a second `Language Model` node to produce the final structured test cases.

Recommended settings:

- `Provider = OpenAI`
- `Model = gpt-4o-mini`
- `Temperature = 0.1`

This keeps the generation consistent and low-hallucination.

### 6. Chat Output

Send the final test case table to `Chat Output`.

## Example Input

```text
POST /api/users - Creates a new user. Required fields: name (string, 2-50 chars), email (valid email format), password (min 8 chars, must include uppercase and number). Returns 201 on success, 400 on validation error, 409 if email exists.
```

```text
TC_001 | Create user with valid name, email, and password | POST /api/users | name=John Doe, email=john@example.com, password=Password1 | 201 | User is created successfully
TC_002 | Reject request when name is missing | POST /api/users | email=john@example.com, password=Password1 | 400 | Validation error for missing required field
TC_003 | Reject request when email format is invalid | POST /api/users | name=John Doe, email=johnexample.com, password=Password1 | 400 | Validation error for invalid email
TC_004 | Reject request when password is shorter than 8 characters | POST /api/users | name=John Doe, email=john@example.com, password=Pass1 | 400 | Validation error for password length
TC_005 | Reject request when password has no uppercase letter | POST /api/users | name=John Doe, email=john@example.com, password=password1 | 400 | Validation error for password rule

```

## Why This Output Works

- step 1 extracts rules before generation starts
- step 2 systematically expands those rules into happy path, negative, and boundary coverage
- the prompt explicitly blocks undocumented assumptions
- the format is submission-friendly and easy to review

## Evaluation Criteria Mapping

### 1. Approach

The workflow uses the correct chained design:

- analyze the API text
- extract requirements
- generate structured test cases from the extracted requirements

### 2. Working Flow

The output from the first `Language Model` is passed into the second prompt step, so both LLM calls are connected and executed in order.

### 3. Output Quality

The second prompt explicitly enforces:

- happy path coverage
- negative scenario coverage
- boundary condition coverage
- no undocumented invention
