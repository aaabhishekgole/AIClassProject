# n8n Flow: Test Case Generator from JIRA Ticket

## Problem Statement

Build an `n8n` workflow that accepts either:

- a JIRA ticket ID, or
- requirement text pasted directly by the user

The workflow should send the requirement to an LLM and generate functional test cases in a structured table format.

## Context

QA engineers receive feature requirements through JIRA during the STLC process. Writing test cases manually for every ticket is slow, so this workflow automates the `Requirement -> Test Case` step and helps the team produce a first draft quickly.

## Workflow Requirements

1. Use a `Chat Trigger` so the user can type a JIRA ticket ID or paste requirement text.
2. Optionally connect a `Jira` node if API credentials are available.
3. Pass the requirement text to an LLM node such as Groq or Ollama.
4. Use the `RICE POT` framework in the system prompt with the role `Senior QA Engineer`.
5. Add this constraint: `Generate ONLY from the provided requirement. If info is missing, write "Needs clarification".`
6. Return the test cases in a table format.

## Practical Build Choice

For the importable workflow JSON, the safest working version is:

```text
Chat Trigger
  -> Set Requirement Input
  -> Basic LLM Chain
  -> Set Chat Response
```

This version works without needing Jira credentials.

If you want Jira lookup later, insert a `Jira` node between `Chat Trigger` and `Basic LLM Chain`, then pass the ticket description into the same prompt.

## Recommended n8n Flow

```text
Chat Trigger
  -> Set Requirement Input
  -> Basic LLM Chain
  -> Set Chat Response
```

Because the final node outputs a `text` field, the `Chat Trigger` can return the result directly when the response mode is `When Last Node Finishes`.

## Node-by-Node Design

### 1. Chat Trigger

Use `Chat Trigger` so a user can paste:

- a full requirement
- a user story
- or a JIRA key such as `AUTH-102`

This is the most natural interaction for a demo because the user can test the workflow directly inside n8n chat.

### 2. Set Requirement Input

Add a `Set` node that copies the incoming chat value into a stable field:

```text
requirement_input = {{$json.chatInput}}
```

This keeps the rest of the workflow consistent and easier to debug.

### 3. Basic LLM Chain

Use a `Basic LLM Chain` with a connected `Groq Chat Model`.

Suggested model:

- `llama-3.3-70b-versatile`

If Groq is unavailable in your environment, you can swap in any accessible free chat model and keep the same prompt.

## Recommended System Prompt

Use this in the `Basic LLM Chain` as the system message:

```text
RICE POT Framework

Role:
You are a Senior QA Engineer.

Input:
You will receive either a software requirement, a user story, acceptance criteria, or a short JIRA-style description.

Context:
Your job is to derive functional test cases from the provided requirement only.

Expectation:
Cover positive, negative, boundary, and error-handling scenarios when they are supported by the provided requirement.

Process:
- Read the requirement carefully.
- Identify the main business rule, validations, limits, and risk areas.
- Generate only test cases that are justified by the given requirement.
- If the input is only a ticket ID and does not contain usable requirement details, do not invent anything.

Output:
Return a markdown table with exactly these columns:
| TC_ID | Test Scenario | Preconditions | Test Steps | Test Data | Expected Result | Priority |

Constraints:
- Generate ONLY from the provided requirement.
- If information is missing, write "Needs clarification".
- Do not invent backend rules, UI details, or validations not present in the requirement.
- Keep test cases concise and functional.
- Produce 6 to 8 test cases when enough detail exists; otherwise produce fewer accurate rows.
```

## Recommended User Prompt

Use this as the prompt text in the chain:

```text
Generate functional test cases from this requirement:

{{$json.requirement_input}}
```

## Example Input

```text
As a user, I should be able to reset my password using my registered email. The system should send a reset link valid for 24 hours. After 3 failed attempts, the account should be locked for 30 minutes.
```

## Example Output

```markdown
| TC_ID | Test Scenario | Preconditions | Test Steps | Test Data | Expected Result | Priority |
| --- | --- | --- | --- | --- | --- | --- |
| TC_001 | Reset password with registered email | User account exists | Open forgot password page, enter registered email, submit | Registered email | Reset link is sent to the registered email | High |
| TC_002 | Reset password with unregistered email | None | Open forgot password page, enter unregistered email, submit | Unregistered email | Needs clarification | Medium |
| TC_003 | Use reset link within validity period | Reset link generated | Open reset link within 24 hours and set new password | Valid reset link | Password is reset successfully | High |
| TC_004 | Use expired reset link after 24 hours | Reset link generated | Open reset link after 24 hours | Expired reset link | System rejects the link and asks for a new reset request | High |
| TC_005 | Lock account after 3 failed attempts | User has a reset flow in progress | Enter incorrect or invalid attempts 3 times | Invalid attempts | Account is locked for 30 minutes | High |
| TC_006 | Retry after lock period expires | Account previously locked | Wait 30 minutes and retry the flow | Registered email | User can continue after lock duration ends | Medium |
```

## Handling Plain JIRA IDs

If the user sends only a value like `AUTH-102` and there is no Jira node in the workflow, the model should return rows with `Needs clarification` instead of fabricating missing requirements.

This keeps the workflow honest and still satisfies the anti-hallucination rule.

## Optional Jira Extension

If Jira credentials are available, extend the workflow like this:

```text
Chat Trigger
  -> Set Requirement Input
  -> Jira
  -> Basic LLM Chain
  -> Set Chat Response
```

Use the `Jira` node to fetch:

- issue summary
- description
- acceptance criteria

Then pass that text into the same LLM prompt.

## Why This Approach Works

- `Chat Trigger` makes the workflow easy to demo.
- `Set` normalizes the input field.
- `Basic LLM Chain` is enough because this is prompt-to-output generation, not tool orchestration.
- the strict prompt prevents hallucinated test cases.
- returning a `text` field lets n8n chat display the response directly.

## Evaluation Criteria Mapping

### 1. Approach

The workflow uses the correct logical flow:

- chat input
- optional normalization
- LLM-based generation
- structured output back to chat

### 2. Working Flow

The importable version works immediately for pasted requirement text and does not depend on Jira credentials.

### 3. Output Quality

The system prompt enforces:

- structured table output
- coverage of major scenarios
- requirement-only generation
- `Needs clarification` for missing information
